## 📝 Claude AI Architect Exam Question

> **Customer Support Resolution Agent**
>
> Production data shows that in 12% of cases, your agent skips `get_customer` entirely and calls `lookup_order` using only the customer's stated name, occasionally leading to misidentified accounts and incorrect refunds. What change would most effectively address this reliability issue?

### ✅ The Answer

> Add a **programmatic prerequisite** that blocks `lookup_order` and `process_refund` calls until `get_customer` has returned a verified customer ID.

### 💡 Why?

Prompt-based constraints or text guidelines (Options A and C) are behavioral suggestions that an LLM can still bypass under high prompt confusion or unexpected user inputs. For critical enterprise risk mitigation (like preventing unauthorized data leaks or misallocated money transactions), you must implement **strict, programmatic validation rules on the application runtime layer** that reject downstream tool calls until the prerequisite verification parameters exist.

## 🏗️ Architecture Walkthrough

### 🔍 The Problem

In 12% of production cases, the AI agent skips the `get_customer` verification step and proceeds directly to `lookup_order` using only the customer's stated name. This causes:

- 🚫 **Misidentified accounts** — Alice's refund goes to Bob's account
- 💸 **Financial leakage** — incorrect refunds to wrong accounts
- ⚠️ **Data exposure** — cross-account information disclosure

### ❌ Act 1 — The Naive Anti-Pattern

```javascript
// ❌ Anti-pattern: get_customer is OPTIONAL
if (withCustomerId) {
  await get_customer(customerName); // Can be SKIPPED!
}
const order = await lookup_order(orderId, null); // No customer ID guard
await process_refund(orderId, order.amount, null); // Wrong account charged
```

The naive approach uses prompt-based guidance only. The LLM can bypass text instructions under high prompt confusion. There is **no runtime enforcement** — the system trusts the model to follow instructions, which fails 12% of the time.

> 🎬 **[Watch the Naive Failure Demo](./remotion/exports/scene2-naive-failure.mp4)**

### ✅ Act 2 — The Resilient Architecture  

```javascript
// ✅ Resilient: get_customer is MANDATORY
const customer = await get_customer(customerName); // ALWAYS called
if (!customer.id) {
  throw new VerificationRequiredError(); // Blocks downstream
}
const order = await lookup_order(orderId, customer.id); // Verified ID
await process_refund(orderId, order.amount, customer.id); // Correct account
```

The resilient approach implements a **programmatic prerequisite** — a strict runtime validation rule that blocks all downstream tool calls until the prerequisite verification parameter (customer ID) exists. The guard executes **before** any destructive operation.

> 🎬 **[Watch the Resilient Success Demo](./remotion/exports/scene4-resilient-success.mp4)**

### 🏛️ Three Pillars of Resilience

| Pillar | Icon | Pattern | Purpose |
|---|---|---|---|
| **Context Fork** | ⏸️ | Sandbox isolation | Downstream tools CANNOT execute without prerequisite data |
| **Validation Shield** | 🛡️ | `if (!customerId) throw Error` | Programmatic guard enforced at runtime — not prompt-based |
| **Token Filter** | 🔍 | Argument hints | Only verified parameters reach downstream calls |

## 🔑 Key Code Snippets

### ❌ Naive — The Vulnerability

```javascript
// src/subagent-naive.js:27
if (withCustomerId) {
  customerData = await get_customer(customerName); // OPTIONAL — 12% skip
}
const order = await lookup_order(orderId, customerId); // null customerId
// ❌ No guard. Refund processed on WRONG account.
```

### ✅ Resilient — The Fix

```javascript
// src/subagent-resilient.js:22
customerData = await get_customer(customerName); // ALWAYS called
customerId = customerData.id;

// ✅ Programmatic prerequisite guard
if (!customerId) {
  return new Result(false, "🛡️ Blocked: Customer ID required before downstream operations");
}
// Only reaches here if customer ID is verified
const order = await lookup_order(orderId, customerId);
```
